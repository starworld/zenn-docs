## 技術者として「これは面白い」と思った理由

最初に技術的な観点での正直な評価を書いておく。

zo.computerの実体は、**Modalというクラウドプロバイダー上に構築された、ユーザーごとの専用Linuxサーバー（コンテナ）**だ。Basicプランで4コア/32GB RAM、100GBのストレージ。インフラとしてはCloudflare（CDN）、Neon（Postgres）、Upstash（Redis）が使われている。

AIとの統合はClaude・GPT・Geminiがネイティブに組み込まれており、BYOKでOpenAIやAnthropicの自前のAPIキーも持ち込める。

ホスティングの技術スタックはHono（Webフレームワーク）+ Bun（ランタイム）+ SQLite（DB）がデフォルトで、zo.spaceというサブドメインで即時公開できる。カスタムドメインはCloudflare経由でTLS証明書が自動発行される。

これを聞いて「ああ、よくあるマネージドサービスか」と思うかもしれない。僕も最初はそう思った。

違ったのは、**「AIとサーバーが本当の意味で統合されている」**点だ。

マネージドVPSにAIのAPIを個別に繋げる構成では、AIとサーバーは「連携している」だけで「統合されている」ではない。Zoの場合、AIがサーバーのファイルシステム・プロセス管理・デプロイ機能に直接アクセスできる。「コードを書いてください」と頼んだら、書くだけでなく実行し、問題があれば修正し、完成したらデプロイするところまで自律的にやる。

これは確かに「次元が違う」と感じた。

## MCPサーバー統合——Claude CodeやCursorとの連携

エンジニア視点で最も注目したのが、MCPサーバー（Model Context Protocol）としての機能だ。

zo.computerは `https://api.zo.computer/mcp`（要認証）でMCPサーバーとして機能する。Claude CodeやCursorのMCPクライアントからZoに接続すると、**Zoのファイルシステム・連携サービス・ツールにClaude CodeやCursorから直接アクセスできる**。

具体的な使い方として、僕がやっていること：

1. **Cursorでコードを書く**：ローカルのCursorからZoのMCPに接続し、Zoサーバー上のファイルを直接編集
2. **変更をZoにデプロイ**：「デプロイして」と言えばZoがそのままzo.spaceに公開
3. **実行結果をZoで確認**：Zoのログ・ファイル管理ダッシュボードで確認

従来の開発フローが「ローカル → GitHub → CI/CD → Vercel」だったとすると、Zoでは「ローカル（Cursor+MCP）→ Zo（ワンコマンドでデプロイ）」に圧縮される。

設定もシンプルで、Claude Codeなら `.mcp.json` にZoのエンドポイントとAPIトークンを追加するだけだ。詳細は公式ドキュメント（docs.zocomputer.com）を参照してほしい。

## SSH接続とコマンド実行——本当のLinuxサーバーとして使う

zo.computerはSSH接続に対応している。つまり、**普通のLinuxサーバーと同じように使える**。

```bash
# Cursorのターミナルから直接Zoに接続
ssh user@zo.computer -i ~/.ssh/zo_key
```

接続後はBash、Python、Node.jsなど通常のLinux環境が使える。GitもnpmもPipもある。自分の好きなエディタで接続して開発できる。

「マネージドサービスだけど、本物のLinux」という感覚は使ってみると快適だ。VPSを使っているエンジニアにとっての学習コストはほぼゼロで移行できる。

## 24時間稼働エージェントによる自動化例

エンジニアとしての具体的な自動化ユースケースをいくつか紹介する。

### コード生成の自動化

「毎週日曜日の夜に、今週のGitHubのコミット履歴を見て、翌週のリファクタリング候補を提案してください」というAgentを作った。GitHub連携で直接コミットを参照し、Markdownレポートとしてストレージに保存される。月曜の朝には「今週改善すべきコードの候補」が出来上がっている。

### デプロイの自動化

「指定のフォルダのファイルが更新されたら、自動でzo.spaceに再デプロイして、完了したらTelegramに通知して」というAgentを設定した。ファイルを保存するだけで自動デプロイが走る。簡易的なCI/CD的な使い方だ。

### ドキュメント自動生成

「コードが更新されたら、関数のJSDocコメントを更新して、READMEの該当箇所も自動更新して」という指示を試みた。精度は100%ではないが、下書きとして十分機能する。

### 定期的なバックアップ

「毎日深夜0時に、ストレージの重要フォルダをzipにまとめて、Google Driveに保存して」というAgentも動かしている。以前は手動でやっていた作業が完全に自動化された。

## zo.spaceへのホスティング——Vercelの代替として

エンジニアがVercelを使う主な用途のひとつは、個人プロジェクトやAPIの手軽な公開だ。zo.computerのzo.spaceはこの用途をカバーできる。

zo.spaceで公開できるもの：
- Reactで書いたWebアプリ
- HonoベースのAPIサーバー
- 静的サイト
- ポートフォリオ・リンク集

公開の手順が「言ったらおわり」という点が特徴的で、「このAPIをzo.spaceで公開して」と言うだけでデプロイが完了する。

Vercelとの比較で正直に言うと、**フロントエンドの洗練度やCDNのパフォーマンスはVercelが上**だ。Next.jsをヘビーに使っている人は当面Vercelから離れなくていいと思う。ただ、個人の小規模プロジェクトや学習用途のAPIサーバーなら、Zoで十分代替できると感じている。

Basicプラン（$18）では最大5つのサービスをホスティングできる（Freeは1つ）。カスタムドメインもCloudflare経由で使える。

## 技術者として注意すべきポイント

正直レビューとして、技術者が気をつけるべき点も書いておく。

**APIトークンの管理**

ZoのAPIトークンはフルアクセス権限で、一度しか表示されない。`.gitignore` に必ず追加して、GitHubにコミットしないこと。これは絶対に守ってほしい。漏洩した場合のリスクが大きい。

**BYOKの設定**

自分のOpenAIやAnthropicのAPIキーをBYOKとして設定すると、Zoのクレジットを消費せずにAPIを使える。ヘビーに使うエンジニアはこの設定をするとコストを抑えられる。

**Freeプランのスリープ**

24hエージェントや常時稼働のAPIサーバーを動かすなら、Basicプラン（$18）が必要だ。Freeプランでサーバーがスリープしていると、スケジュール実行が動かない。

**Gmail連携のリスク**

公式も警告しているが、Gmail連携はプロンプトインジェクションのリスクがある。最初はRead Onlyから始めて、メール送信権限は慎重に付与してほしい。

## 「集約」してわかったこと

ツールを一カ所に集約して気づいたのは、**「データが一カ所に集まることのAI活用効果」**だ。

以前はGitHubにコード、Dropboxに資料、VPSにサービスが分散していた。AIに「先月のコードと今月の資料を見比べて改善点を提案して」と頼んでも、それぞれのサービスにまたがって情報を集めるところから始まる。

Zoに集約すると、これが自然にできる。「ストレージの中のすべてを参照しながら考えて」という指示に対して、AIがファイルシステム全体をコンテキストとして活用できる。これは使ってみて初めてわかった価値だった。

技術者にとって、zo.computerは「VPSとAIと自動化を組み合わせたかった人が待っていたもの」に近い。設定不要でその環境が手に入る。
